# ADR-046 Phase 2 实施计划 —— 多行 claim / 多行 report

> 状态:**计划待开工**(Phase 1 已全端到端落地 #613–627;Phase 2 按 ADR gate「命中上万 fan-out 再上」)。
> 权威设计源:[`docs/architecture/adr/ADR-046-file-bundle-aggregation.md`](../architecture/adr/ADR-046-file-bundle-aggregation.md) §实施分阶段 / §已决。
> 本文是该 Phase 2 的逐切片实施清单,先定语义再写码。

## 范围一句话

把 orchestrator 的 claim/report 从「一次领/报 1 个 partition」加一个**批量变体**「一次事务领/报 K 个独立 partition + outbox 批写」,控制面往返 O(N)→O(N/K)。**仅此一项优化**,partition 语义/状态机/幂等/lease/worker 执行全不变。

## 护栏(不可越)

- **纯加法**:新增批量变体端点,**单 partition 的 claim/report 路径一字不动**;`POST /tasks/leases/renew-batch`(`TaskLeaseRenewBatchRequest/Response`)是现成范式,照抄其形态。
- **无束状态机、无 worker 束循环**(2026-06-21 评审已砍的重方案,不得滑回)。
- **opt-in flag 默认关**(`batch.task.batch-claim.enabled`,可按 tenant/job 覆盖),先只对 `BUNDLE_*` 作业开。
- **不碰 DB schema**(partition 绑定列 Phase 1 已加)、**不碰 partition UNIQUE/PK**(不动 `ON CONFLICT` 幂等契约)。

## 关键代码锚点(现状)

- `batch-orchestrator` `controller/TaskController`:`POST /{taskId}/claim`、`POST /{taskId}/report`、已有 `POST /tasks/leases/renew-batch`(批量先例)。
- `application/service/task/TaskControllerApplicationService`:`claim` / `report`(report 已带 5 次 + random jitter 退避,防 Citus 并发 report 分布式死锁)。
- `application/service/task/DefaultPartitionLifecycleService`:`claimPartition`(经 `jobPartitionMapper.claimPartition`,`int updated` 乐观锁 CAS)、`renewLease`、`reclaimExpiredPartitions`。

## 切片(逐 PR,每片可独立合、可回退)

### 2.0 基线 + flag(低风险先行)
- 加 `batch.task.batch-claim.enabled` 开关 + per-job 覆盖;默认关。
- 搭上万 fan-out 压测脚本(复用现有 sim),量 O(N) claim/report 往返与锁争用,作为 Phase 2 验收对照基线。**没有基线数不进 2.1。**

### 2.1 多行 claim(orchestrator)
- 新端点 `POST /tasks/claim-batch`(仿 renew-batch):`TaskClaimBatchRequest{tenantId, workerId, taskIds[]}` → `List<EffectiveTaskConfig>`(只含**认领成功**的,已被别人领的跳过)。
- app service `claimBatch`:逐 taskId 走**现有** `claimPartition` CAS(best-effort 独立),**部分成功是正常语义**,返回赢到的子集;幂等沿用 per-partition `partitionInvocationId`。
- 一条 UPDATE 批量认领(`WHERE id IN (...) AND status='READY'` + RETURNING)是后续优化;**先用循环跑通语义**。
- OpenAPI / protocol.md 同 PR 更新(若端点对外;内部 worker 端点按 internal 约定)。

### 2.2 多行 report(orchestrator)—— 最难的一片
- 新端点 `POST /tasks/report-batch`:`List<TaskExecutionReportDto>` → **逐项结果** `[{taskId, ok|CONFLICT|...}]`。
- **批内部分失败是核心**:K 个里某项版本 CAS 撞并发,**必须提交其余 K-1、单独退冲突项**,worker 仅重报冲突的,**不得整批回滚**。
  - 取舍写死:**逐项 savepoint**(同事务内批量 outbox + 任一项失败只回滚该项 savepoint) vs 逐项 `REQUIRES_NEW`(隔离强但 outbox 不能批写)。**推荐 savepoint**,保住「K 个 outbox 一次批写」的省 churn 收益。
  - 复用 report 的 5 次 + jitter 退避,作用在**整批**层。
- outbox 批写:K 个事件一次 INSERT(`outbox_event` 同事务,主链约束不破)。

### 2.3 worker-core 消费端攒批
- 消费 node-direct 派单 topic 时,小窗口(`size≤K 或 ≤T ms`)攒多条 → 一次 `claim-batch` → **逐 partition 按现有方式独立处理**(per-partition try/catch、checkpoint、各自 lease 续租用 `renew-batch`)→ 一次 `report-batch`。
- `<2` 条**回退单条路径**(零额外开销);offset **仅在 report-batch 确认后**提交(守 at-least-once)。

### 2.4 观测
- batch size 直方图、churn O(N)→O(N/K) 实测指标、批内 CONFLICT 计数;监控页下钻不变(复用 job_instance→partition)。

### 2.5 IT + 压测验收门(不过不合)
- IT:批内单项 CAS 冲突(其余照常推进)、批量 report 重放幂等、worker 批处理中途 crash 恢复、混表异构批。
- 压测:上万 fan-out 跑 2.1+2.2+2.3,对照 2.0 基线证明 claim/report 争用实降且**零正确性回归**;达不到收益就不启用 flag。

## 风险与缓解

| 风险 | 缓解 |
|---|---|
| 动最敏感热路径(CAS/幂等/lease/同事务 outbox) | 全程新变体 + flag 默认关,单 partition 路径不动;可秒回退 |
| 批内部分失败语义错 → 丢/重推 partition | 2.2 savepoint 逐项 + 返回逐项结果 + 专项 IT |
| Kafka 攒批破 at-least-once | offset 仅 report-batch 后提交;`<2` 回退单条 |
| 收益不达预期(低千 Phase 1 已够) | 2.0 基线先量,无收益不启用 —— 本就是 YAGNI gate |

## 顺序与体量

`2.0 →(2.1 ∥ 2.2)→ 2.3 → 2.4 → 2.5`。2.2 是大头(部分失败语义 + savepoint + 批 outbox),其余偏机械。**约 5 个 PR**,每片纯加法可单独回退。
