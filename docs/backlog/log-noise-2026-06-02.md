## 1) [HIGH] worker-export 反复 BadSqlGrammar `biz.export_batch`  — 日志已降噪 ✅(根因待处理)

**现象**
```
WARN  c.e.b.worker.exports.stage.GenerateStep - catch:Exception:
  BadSqlGrammarException: PreparedStatementCallback; bad SQL grammar
  [SELECT "id","batch_no","total_count" FROM "biz"."export_batch"
   WHERE "tenant_id" = ? AND "batch_no" = ? LIMIT 1]
INFO  AbstractTaskConsumer - EXPORT task processed: taskId=216, success=false
```
每 ~90s 一次,固定同一 taskId=216,租户 ta。

**根因**
sql_template_export 配置指向业务库 `biz.export_batch`,而该业务库未建此表;
任务被上游 trigger(见 #3)持续重投。

**关联**
- orchestrator 同一时刻产出 dead letter:`tenantId=ta, partitionId=216, errorClass=SYSTEM`
- 上游 trigger 链:#3 的 misfire 补点 → orchestrator 派发 → export worker 失败

**建议处理**
- 短期:把 instanceNo `inst-20260529T142430Z-117e582e` 在 console 手动 TERMINATE,或调小重试上限让其早进死信
- 中期:校验该 sql_template_export 配置的 targetTable 是否引用了不存在的 biz 表(V66 的 biz_table_schema 快照本来就是为这种校验设计的,看是否漏挂)
- 长期:bad SQL grammar 这类"配置错"应在 console 上传/保存时拦,而不是 runtime 才暴露

## 2) [MED] orchestrator WORKFLOW 类型 worker 无候选 — 日志已降噪 ✅(根因待处理)

- 如果该任务本就不该跑:从 ta 租户的 workflow 配置里下线
- 期间想消音:`DefaultWorkerSelector` 增加节流(同 (tenant, group) 30s 内只 WARN 一次)

## 3) [MED] trigger TA_PROCESS_CUSTOMER_SUMMARY misfire 补点风暴

**现象**
```
INFO  HashedWheelTriggerScheduler - misfire detected:
  job=TA_PROCESS_CUSTOMER_SUMMARY
  scheduledFireTime=2026-06-01T15:54:00Z policy=NONE
```
每分钟一次,scheduledFireTime 单调推进,正在追 10+ 小时前的 backlog。

**根因**
昨日服务停过一段时间,该 job 是 1min 周期,misfire policy=NONE 意为"每个错过时刻都补一次",
导致正在以"每分钟 1 个"的速度倒带回补。

**建议处理**
- 立刻止血:把该 job 的 misfire policy 改为 `LATEST_ONLY` 或 `IGNORE_ALL_MISSED`
- 该 job 本就是 #1 export 失败的源头,补点风暴 → 报错风暴
- 检视其他 job 的 misfire policy,凡是"补点无业务价值"的高频 job 都不应是 NONE/全补

## 4) [LOW] console SSE ticket 幂等键复用

**现象**
```
WARN  ConsoleIdempotencyInterceptor - duplicate idempotency key rejected
  (already done): key=0fd44a66-...,
  uri=/api/console/auth/stream/ticket, tenant=tb
```
每 5s 一次,FE(batch-console)固定一个 idempotency-key 在轮询 SSE auth ticket。

**根因**
FE 在请求 `/auth/stream/ticket` 时复用同一个 Idempotency-Key(很可能是 axios 拦截器统一注入,
而 ticket 这种"每次都要新值"的接口没排除)。BE 正确拦下,但每 5s 一条 WARN 太吵。

**建议处理**
- FE:`/auth/stream/ticket` 调用前清空/重置 Idempotency-Key(或对该 URL 路径白名单跳过统一注入)
- BE:`/auth/stream/ticket` 这种"每次结果都不同"的接口本不应进 Idempotency 拦截器,
  从拦截 URL 名单里排除即可,顺便消音
