## 2. 修复的 BE bug(已写代码)

### 2.1 `ImportIngressScanner` 单租户硬绑(M+)

**问题**:Scanner 启动时读 `workerConfiguration.tenantId()` 单一 tenant,所有 MinIO 对象登记的 file_record 都打这一个 tenantId。多租户场景无法用一个 worker-import 同时服务 ta/tb/tc。

**修复**:`batch-worker-import/.../runtime/ImportIngressScanner.java` 加 `resolveTenantFromObjectName(objectName)` — 按 MinIO prefix 解析 tenantId(`ingress/ta/foo.csv` → `ta`),解析不到时 fallback worker 自身 tenantId(保持旧布局兼容)。`tryRegister` 内 5 处 `workerConfiguration.tenantId()` 全替换为 `resolvedTenant`。

**验证**:投放 `ingress/{ta,tb,tc}/*-20260529.csv` → scanner 30s 后登记 3 个 file_record,各自带正确 tenant_id。

### 2.2 `GenerateStep.toStringKeyMap` 不支持 PG jsonb PGobject(L)

**问题**:`batch-worker-export/.../stage/GenerateStep.java` 的 `toStringKeyMap(Object)` 只识别 `Map` 和 `String`,jsonb 列经 MyBatis 返回 `org.postgresql.util.PGobject` 时 fallthrough 到 `Map.of()`,导致 `query_param_schema.export_data_ref` 永远读不到 → `export_data_ref is required` 误报。

**修复**:加 `PostgresqlJsonbTexts.tryExtract(raw)` 兜底反射抽 PGobject 的 JSON String(跟 `JdbcMappedImportSpec.toMap` 对齐)。

**验证**:EXPORT 立即跑通,MinIO outbound 落 csv。

### 2.3 fixture 约定:WorkflowNodeCode 严格匹配 START/END(M)

**问题**:`WorkflowNodeCode` 枚举只有 `START` / `END`(无 NODE_ 前缀)。DAG `resolveInitialNodes` 查 `workflow_edge.from_node_code = "START"`,fixture 写 `NODE_START` 时找不到 outgoing edge → initialNodes 空 → fallthrough 到 plan dispatch → workerGroup=WORKFLOW 选 worker 失败 → workflow 卡 CREATED。

**修复**:fixture 约定 node_code 必须严格 `START` / `END`(其他节点可自由命名)。

**验证**:改 SQL 节点 code 后,workflow_run 立即 SUCCESS,3 个 node_run 持久化,子 instance 自动 spawn。

**建议**:`PackageValidator` 加 fixture 校验,显式拒绝非 START/END 命名的开始/结束节点。

## 3. fixture 设计层面发现(非 bug,工程缺口)

| 项 | 现状 | 修法 |
|---|---|---|
| `pipeline_step_definition.step_params` | fixture 几乎全 `{}` | 真实业务参数应在 file_template_config(jdbcMappedImport / sql_template_export 都已支持) |
| TA_IMPORT_CUSTOMER 缺 PREPROCESS step | RECEIVE → PARSE 直跳 → file_state RECEIVED → PARSED 状态机禁止 | 加 PREPROCESS step,负责 RECEIVED → PARSING 转换 |
| TB_EXPORT_STATEMENT 只有 1 个 GENERATE step | 缺 PREPARE → templateConfig 没加载到 context → GenerateStep 找不到 export_data_ref | EXPORT pipeline 5-step 是必须 |
| job_definition.execution_mode | importer 不识别 column,导致 NOT NULL violation | importer 加 column 映射,或 SQL UPDATE 默认 'FULL' |
| EXPORT `default_query_sql` 必须含 `:batchNo` | `sql_template_export` validator 要求 named param | SQL 写 `WHERE tenant_id = :tenantId AND :batchNo IS NOT NULL` |
| ta/tb/tc fixture workflow_node 用 NODE_START | DAG 找不到 outgoing edge | 改 fixture xlsx 或加 PackageValidator 拦截 |
| ta channel_config SFTP 用 `host` key | worker 代码读 `sftp_host` key | 统一 key 命名(channel JSON 配置约定) |

## 6. 已知坑

### 6.1 orchestrator 首启 NoClassDef

PG migration 未完成时启动 orch → Mapper Inner class load 失败 → KafkaListener 死。**症状**:`outbox_event does not exist`、`worker_registry does not exist`。**解决**:确认 docker postgres healthy 后,执行 `bash scripts/local/restart.sh orchestrator`。

### 6.2 fix 后的代码已写、未 commit

- `batch-worker-import/.../ImportIngressScanner.java`(多租户)
- `batch-worker-export/.../GenerateStep.java`(PG jsonb)

后续如 commit,建议:
- Scanner 多租户:加 `batch.worker.import.scanner.multi-tenant-mode` 开关,默认 false 保旧行为
- PG jsonb:无需开关(纯 bugfix)

### 6.3 worker-process plugin id 是 camelCase

`SqlTransformComputePlugin.PLUGIN_ID = "sqlTransformCompute"` — fixture / SQL 配 `impl_code` 时务必 camelCase,**不是** `sql_transform_compute`。

### 6.4 worker-dispatch SFTP/HTTP channel key 命名

| channel_type | config_json 必备 key | 注 |
|---|---|---|
| SFTP | `sftp_host` / `sftp_port` / `sftp_username` / `sftp_password` / `sftp_remote_path` | 不是 `host`/`port` |
| API_PUSH | `url` / `method` / `tokenHeader` | 这三个会被 `ChannelConfigMerge` 白名单忽略 → 但 dispatch 时仍读 `url` |

### 6.5 workflow_node 强约定

- `node_code = "START"` `node_type = "START"` (强匹配 enum)
- `node_code = "END"` `node_type = "END"`
- `workflow_edge.from_node_code = "START"` (不是 NODE_START)
- workflow job_definition `worker_group = NULL`(orchestrator 自管 DAG)
- `dag_enabled = true`
