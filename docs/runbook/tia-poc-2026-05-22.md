# Test Impact Analysis (TIA) POC — 测试类粒度

日期:2026-05-22
状态:**POC,不接入 pr-gate.yml**,人工 review 后决定是否启用。

## 1. 背景

现 `pr-gate.yml` 的 "Detect affected scope" 步骤是 **module 粒度**:
- 看 `git diff` 改了哪些目录,映射到 9 个 `batch-*` module
- 选中的 module 全跑(全部测试),其它 module skip
- 改 `batch-common/*` / `pom.xml` / `.github/workflows/*` 走 `full` 兜底,全 9 模块全跑

问题:即使 partial,中等大小的模块也会触发上百个测试。473 个 test class 里很多是「调用链上没碰」的。

目标:**class 粒度 TIA** — PR gate 只跑「调用链 touch 到被改文件」的测试。

## 2. 选型

### 候选 A:`io.github.teyckmans:test-impact-analysis-maven-plugin`(放弃)

业界 maven 原生 TIA 主流候选。基于 JaCoCo coverage 数据 + git diff,反查测试。

验证(2026-05-22):
- `https://repo1.maven.org/maven2/io/github/teyckmans/test-impact-analysis-maven-plugin/maven-metadata.xml` → **HTTP 404**
- `central.sonatype.com` 搜索无结果
- `github.com/teyckmans/test-impact-analysis-maven-plugin` → **HTTP 404**(仓库不存在或已删除)

结论:Maven Central **未发布**,GitHub 仓库**已不存在**,无法引入。

### 候选 B:Gradle Enterprise / Develocity Predictive Test Selection(放弃)

商业方案,要 Gradle Enterprise / Develocity license。本项目 Maven multi-module 且无 license。

### 选 C:自建 git diff + 静态依赖图(采用)

`scripts/ci/select-affected-tests.py`(178 行):
1. `git diff <base>..<head>` 拿改动 `.java` 文件
2. 解析 `package` + 文件名 → FQN
3. 索引所有 main 源(1573 个)的 `import` 列表
4. 反向求 5 跳传递闭包:谁 import 了 changed FQN → 加入 impacted 集
5. 同 package 直接用 simple name(无 import)的情况:扫文本字界匹配
6. 扫所有 486 个测试源,同样规则,引用任一 impacted FQN 即选中

理由:
- **零外部依赖**(只用 stdlib + git)
- 不依赖运行时 coverage 数据(后者 cold-start 没数据,首次 PR 必 fallback)
- 静态保守 — 倾向多选不漏选(safe over-approximation)

## 4. 本地验证结果

(执行机:本地 git diff,base=HEAD~1)

| Scenario | 改动文件 | impacted main FQN | selected tests | 对比 486 总 |
|---|---|---|---|---|
| S1 service impl | `DefaultTaskAssignmentService.java` | 1 (hop1=0) | **1** | 99.8% ↓ |
| S2 interface | `TaskAssignmentService.java` | 23 (hop1→5: +3/+7/+6/+4/+2) | **19** | 96.1% ↓ |
| S3 controller | `ConsoleMetaController.java` | 1 (hop1=0) | **1** | 99.8% ↓ |
| S4 common util | `Texts.java` | 517 (hop1→5: +139/+154/+122/+69/+32) | **379** | 22% ↓ |

观察:
- 改 leaf 类(controller / 单实现 service)→ 选中极少 → 与 PR 测试范围 1:1 匹配
- 改 interface → 中等(19)— 因为传递 hop 把所有调用方拉进 impacted
- 改 batch-common widely-used util → 大半测试都选中,接近原 `full` 兜底
  - **这条对齐了现有 `Detect affected scope` 的设计:`batch-common/*` 改动走 `full`**

## 6. 未覆盖盲区(必须 fallback)

非 Java 改动 — TIA 直接输出 0 tests,**必须 fallback 全跑**:

| 类型 | 例子 | 影响范围 |
|---|---|---|
| MyBatis XML | `*Mapper.xml` | SQL 改了但 Java mapper 没改 → 漏掉 |
| Flyway migration | `db/migration/V*.sql` | schema drift 漏检 |
| 配置 properties | `application*.yml` / `*.properties` | Spring boot config 漂移漏检 |
| Helm / docker | `helm/**` / `Dockerfile` | 部署相关,本来也不影响单测 |
| 根 `pom.xml` | 版本 / 插件 | 改一行就可能影响全仓 |
| 注解元数据 | annotation processor 生成的代码 | TIA 不解析 |
| 反射调用 | `Class.forName(...)` / SPI | TIA 看不到 |
| Spring DI 注入 | 改 impl,只有 interface 被 import | hop1 = 0,**漏选 IT 测试**(S1 验证已暴露) |

**最后一条最危险**:S1 改 `DefaultTaskAssignmentService` 只选了 1 个直接测试,但若有 IT 通过 `@Autowired TaskAssignmentService` 注入,这些 IT 不会被选中。**这是静态分析的天然边界**。

## 7. 上线建议(开关式接入)

**不要直接替换** `Detect affected scope`。采用三阶段灰度:

### Phase 1(2 周):**影子模式**
- 在 `pr-gate.yml` 增加一个 step,跑 `select-affected-tests.sh` 但**只 echo 到 STEP_SUMMARY**,不影响实际测试范围
- 现有 module 粒度 scope 不变
- 目的:观察「TIA selected」vs「实际全跑里失败的 test」对比,若 TIA 漏掉了失败的 test,记录为 false-negative

### Phase 2(2 周):**双跑校验**
- TIA 选出测试集 A,module 粒度选出测试集 B
- 实际跑 B(保持现状),并行另起 job 跑 A
- 比较两者结果是否一致;A 失败 / B 通过 = bug;A 通过 / B 失败 = TIA 漏选
- 0 false-negative 持续 2 周 → 进入 Phase 3

### Phase 3:**TIA 作为 primary,module 粒度 fallback**
- PR gate primary 跑 TIA(`mvn verify -Dtest="$AFFECTED_TESTS" -DfailIfNoTests=false`)
- TIA 输出为空(非 java 改动)→ fallback 到现有 module 粒度
- module 粒度也 full → 全跑兜底
- **保留 `batch-common/*` / `pom.xml` / `.mvn/*` / `.github/workflows/*` 走 full 的规则**(TIA 不要管这些)

### 不上线场景
- main 分支 push:`full-ci-gate` 总跑全集,不动
- release 分支 / tag:总跑全集,不动
- workflow_dispatch 手动:全跑兜底(现 `Detect affected scope` 已是)

## 8. 风险

| 风险 | 缓解 |
|---|---|
| DI / 反射 / SPI 漏选 IT | Phase 1-2 双跑校验,若发现 false-negative > 5%,降级到 module 粒度 |
| TIA 脚本 bug 选 0 个 | `|| mvn verify` fallback;`-DfailIfNoTests=false` 保证空集不挂 |
| `git diff base..head` 在 force-push 后失效 | 现 `Detect affected scope` 已处理(`BASE_SHA` 缺失走 full),沿用 |
| Python 3 不可用 | runners 是 ubuntu-latest 自带 python3,无风险 |
| 测试源数量增长 → 扫描慢 | 当前 486 个测试 / 1573 个 main 源,扫描 < 5s;1 万级再说 |

## 9. 后续

- 若 Phase 1 影子模式 2 周里 TIA 总比 module 粒度小 50%+ → 收益清晰,推进 Phase 2
- 若 TIA 与 module 粒度接近(常见 `batch-common` 改动占多数)→ 收益有限,**放弃**,继续用 module 粒度
- 不接受方案:**强行减小测试范围**(去掉同 package fallback / hop 数砍到 2)— 漏检风险压过 CI 时间收益
