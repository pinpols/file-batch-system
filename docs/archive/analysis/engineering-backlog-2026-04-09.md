# 批量调度系统后续开发清单

## 优先级 P0：先解决生产风险

- 统一启动与迁移链路，消除 `schema.sql` 与 Flyway（`batch-orchestrator/db/migration`）职责重叠问题。
- 移除仓库里的硬编码敏感信息，尤其是 API key、密码、密钥等，统一改为环境变量或密钥注入。
- 增加启动自检与健康检查，启动时能明确告诉你缺哪张表、哪条 migration 没跑、哪个 schema 不一致。
- 强化关键链路幂等性，重点是 `launch`、`outbox`、`partition claim`、`retry`、`late event`，避免重复投递和重复执行。

## 优先级 P1：补齐生产化能力

- 增加全局调度治理能力，包括租户级配额、并发上限、限流、优先级、熔断和公平调度。
- 补齐故障恢复工具，包括任务重放、补偿触发、死信回放、人工修复接口和管理脚本。
- 完善 `SLA`、`late arrival`、`batch day` 的状态机和审计链路，确保每次状态变更都能追溯原因和操作者。
- 强化 worker 生命周期管理，包括心跳、`drain`、强制下线、任务回收、超时中断、孤儿任务清理。

## 优先级 P2：提升工程质量

- 收敛配置结构，统一 `@ConfigurationProperties` 的命名、层次和分布，减少重复但保持清晰边界。
- 给关键 SQL 迁移与初始化逻辑补测试，重点覆盖空库初始化、已有库升级、重复执行幂等性。
- 增加集成测试，至少覆盖 orchestrator 启动、Flyway migration、launch/finish/retry、worker claim/report、outbox 到 MQ 的完整链路。
- 统一日志、指标、trace，做到一条任务实例能完整追踪到数据库、消息队列、worker、补偿和最终状态。

## 如果先做最有价值的 3 件事

1. 统一数据库初始化和迁移。
2. 去掉仓库里的敏感配置。
3. 补 `launch` / `outbox` / `retry` / `worker claim` 的幂等与恢复测试。

## 建议的实施顺序

1. 先修启动和迁移问题，确保所有环境都能稳定启动。
2. 再补幂等、恢复和故障治理，降低生产事故概率。
3. 最后做调度公平性、租户隔离、治理能力和可观测性增强。
