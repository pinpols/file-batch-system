# ADR-012 + ADR-021..027 优先级 + 范围边界纪律 — 2026-05-06

> **范围**：批量系统 design reserve 中 ADR-012（失败分类）+ ADR-021..027（对账 / 取证 / 多日历 / 冷热分层 / 静态校验 / dry-run / 资源亲和）的"做不做 / 什么时候做 / 边界在哪"决策档。
>
> **目的**：把"批量调度 + 文件交付闭环"的系统定位写死，避免 ADR 实施期被诉求拽偏成"企业数据治理平台 + K8s 调度器 + 合规调查系统"。

---

## 1. 系统定位（一句话）

> **本批量系统负责"批量运行控制面 + 文件 / 任务交付闭环"，不扩张成"企业数据治理 + 容器资源编排 + 合规审计平台"。**

| 系统应做 | 系统不应做 |
|---|---|
| 什么时候跑 / 跑哪个批次 | 业务含义最终裁定 |
| 按哪个业务日跑 / 怎么切分 | 财务账务核算 |
| 谁来跑 / 怎么调度 | 全域数据治理 |
| 失败怎么处理 / 结果怎么记录 | 底层容器 / 节点资源编排 |
| 文件怎么收发 / 是否完成交付 | 企业审计调查平台 |
| 是否能追溯 / 是否能取证 | 统一日志平台 / SIEM |
| 配置发布前是否合法 / 调度计划是否合理 | 全链路 APM |

## 2. 优先级矩阵（实施顺序）

| 阶段 | ADR | 决定 | 优先级 | 是否越界风险 | 一句话边界 |
|---|---|---|---|---|---|
| **第 1 阶段（必做）** | **ADR-025** Workflow 静态校验 | 直接做 | **P0** | 无 | 发布前防事故，ROI 高 |
| **第 1 阶段（必做）** | **ADR-012** 失败分类 | 直接做 | **P0** | 无 | 7 enum 封闭，支撑重试 / 告警 / DLQ |
| **第 1 阶段（必做）** | **ADR-023** 多日历联动 + 半天 | 直接做 | **P0/P1** | 无 | 调度核心，不做全球日历 SaaS |
| **第 2 阶段（应做但收敛）** | **ADR-021** 数据对账闭环 | 做，收敛 | **P0/P1** | **中** 容易扩大 | 只做批量交付对账，不做数据治理平台 |
| **第 2 阶段（应做但收敛）** | **ADR-026** dry-run 模式 | 做轻量版 | **P1/P2** | **中** 容易复杂 | 只做计划 / 校验 / Explain，不做副作用沙箱 |
| **第 2 阶段（应做但收敛）** | **ADR-022** Forensic 取证 | 做简化版 | **P1** | **低** 容易变重 | 只做"批次取证包"，不做合规调查平台 |
| **第 3 阶段（暂缓）** | **ADR-024** 冷热分层 + OSS | 暂缓 | **P2** | 无 | 数据量到阈值再做 |
| **第 3 阶段（暂缓）** | **ADR-027** 资源亲和 / 地理 | 暂缓 | **P2/P3** | **高** 容易越界 | 不做 K8s scheduler，多机房之前先靠 workerType |

排序逻辑：

- 第 1 阶段三件套同时改善"发布安全 / 失败治理 / 调度严谨"，相互独立可并行落地；
- 第 2 阶段三件套都是"价值高但易扩张"，必须靠各自 ADR 内 boundary 段长期停滞边界；
- 第 3 阶段两件是"规模 / 部署形态决定的"，触发条件不到不开工。

## 3. 各 ADR 的边界纪律

### ADR-012 失败分类（P0，不越界）

✅ **做**：
- `FailureClass` 7 enum 一等公民（INFRASTRUCTURE / DATA_QUALITY / BUSINESS_RULE / CONFIG / UPSTREAM_DELAY / TIMEOUT / UNKNOWN）；
- 按 class 派发 retry / escalation alert routing / 自动 RERUN；
- worker SDK 显式上报 + classifier chain 回退。

❌ **不做**：
- 让 worker 自定义新 FailureClass（枚举封闭，扩展走新 ADR）；
- 把 FailureClass 拆成多个终态（保留 FAILED 单一终态 + class 列分维度）；
- 让 ops 在 console 手填 errorClassHint 改分类（污染审计；reclassify 走独立审批 API v2）；
- 按 class 分流到不同 worker pool（与 ADR-027 资源亲和正交，v1 不做）。

---

### ADR-021 数据对账闭环（P0/P1，易扩大边界）

✅ **做**：
- 4 类规则（ROW_LEVEL / TABLE_LEVEL / CROSS_TABLE / CROSS_DAY）；
- BLOCKER / WARN / INFO 三档 severity；
- DQ gate 嵌入 ADR-017 result_version EFFECTIVE 链路，DQ 失败不让 promote；
- DQ 失败 → ADR-012 的 DATA_QUALITY failure class，不重试。

❌ **不做（边界红线）**：
- 主数据治理 / 数据血缘 / 全域数据质量平台；
- 财务总账核算规则 / 风控量化裁定；
- 跨系统业务语义仲裁；
- DQ 失败"自动修复"（auto-fix 反保守原则）；
- BLOCKER 规则一键禁用 ops 后门（要绕过必经 MANUAL_APPROVAL 留痕）；
- 跨租户共享规则（按租户隔离 + 提供"模板库"做平台级最佳实践）。

**判定边界的提问**：「这条规则的失败结果，是修业务数据还是裁定业务对错？」修业务数据 → 属本 ADR；裁定业务对错 → 不属本 ADR。

---

### ADR-022 Forensic 一键取证（P1，简化版）

✅ **做**：
- 配置类 *_history 影子表（calendar / window / quota / approval / workflow definition）；
- 一键 `POST /api/console/forensic/export {tenantId, bizDate}` → bundle zip（manifest.json + job-definition / instance / step / partition / files / retries / state-transitions / audit / log-index）；
- 7 年 OSS 对象锁保留（compliance mode）+ sha256 attestation。

❌ **不做（边界红线）**：
- 完整合规调查平台 / 法律证据保全工作流；
- 接 Splunk / SIEM 实时流（forensic 是"按需 pull"，不是"持续 push"）；
- 跨系统统一日志平台 / 日志湖；
- v1 不做交互式历史浏览 UI（浏览成本高，console 现有表够 pull）；
- 为运行态表加 *_history（job_instance / partition / task 已有 status 流转 audit log，不双写）。

**判定边界的提问**：「这次取证要回答的问题，能不能靠"一个 bizDate 范围 + 一个 tenantId"圈出来？」是 → 属本 ADR；不是（要跨系统全量调查 / 要追责法律） → 不属本 ADR。

---

### ADR-023 多日历联动 + 半天工作日（P0/P1，不越界）

✅ **做**：
- `calendar_dependency`：A SETTLED 才起 B（中港美串联）；
- `cutoff_schedule` JSONB：圣诞夜 / 春节前夕半天 cutoff、夏季周五早 cutoff；
- `calendar_group` + holiday `scope=GROUP` 共享假日；
- `disaster_day_override`：突发停业 SKIP / DEFER，不改 holiday 表保审计清晰。

❌ **不做（边界红线）**：
- 引入 ICU / iCal RFC 5545 RRULE（过度工程）；
- calendar 自动同步外部公假日 API（数据来源不可控、合规风险）；
- 全球日历 SaaS / "calendar fork"；
- calendar_group 嵌套（flat group 即可）；
- disaster override 未来生效（只解当下灾难，未来停业走正常 holiday）。

---

### ADR-024 冷热分层 + 长保留（P2，暂缓）

✅ **触发条件到了再做**：
1. `archive.job_instance_archive` 行数 ≥ 5 亿 / 单表 ≥ 500 GB；
2. 监管合规要求 ≥ 5 年保留；
3. archive 表查询 P99 > 5 秒持续 14 天；
4. PG 全量 dump 时间 > 4 小时。

❌ **当前先做的轻量版（不属本 ADR，是基线）**：
- 所有大表带 `business_date` / `created_at` 列 + 合理索引；
- 日志类表配保留天数（已有）；
- 删除前写 audit log；
- *预留* `archive_storage_metadata` 列名 / partition 切分键 = `biz_date`（防未来改不动）。

❌ **绝不做**：
- 完整数据湖架构（Iceberg / Delta Lake，过度工程）；
- TimescaleDB / ClickHouse 双引擎（运维多一套）；
- v1 不做 OSS 跨区域复制（DR 场景另算）；
- 冷数据"按需重新 attach 回 PG"（重新加载成本 > 直接查 OSS）。

---

### ADR-025 Workflow 静态校验（P0，必做）

✅ **做**：
- V1-V15 校验项（拓扑 / 不可达 / DSL nodeCode / 跨日依赖 offset & 90 天上限 / OPTIONAL 传染性退化 / GATEWAY ALL_OF & N_OF_M 一致性 / param_schema 类型 / step_registry output contract）；
- enable 切换时同步 + 错误清单一次性返回；
- daily reconciler 重 evaluate 并发 alert（不修复，只告警）。

❌ **不做（边界红线）**：
- 完整 type system（Dagster 风格）—— Java SPI 注解已够用；
- 隐式 type-coercion（int ↔ long）—— 不匹配就是错；
- launch 时重跑 validator（性能不可接受，只在 enable）；
- ERROR 通过 `force=true` 强制 enable 后门（不留逃生路径）；
- disable 路径校验（disable 是清理，不该 block）。

---

### ADR-026 dry-run 模式（P1/P2，轻量版）

✅ **做（三层粒度）**：

| 层 | 粒度 | 行为 |
|---|---|---|
| **L1 CONFIG_VALIDATE** | workflow / job 定义 | 解析 cron / 业务日 / 参数 / DAG / fileTemplate / SQL；只读 + 不调外 |
| **L2 SCHEDULE_PLAN** | bizDate 维度 | 给定 bizDate，输出"会触发哪些 job / 预计 job_instance 数 / 预计分区数 / 预计 Worker 类型 / 预计输入输出文件" |
| **L3 EXECUTION_PLAN** | task 级 | SQL explain / 文件路径解析 / 模板字段检查 / MinIO key 预生成 / 下游 endpoint reachability |

✅ **配套**：
- `dry_run` 一等字段贯穿 LaunchRequest → instance / partition / task；
- worker SDK `DryRunGuard` 拦所有写业务表 / 外部 IO；
- result_version 不进 EFFECTIVE 链；终态 `SUCCESS_DRY_RUN` / `FAILED_DRY_RUN` 隔离。

❌ **不做（边界红线）**：
- "FULL_SIMULATION" 模式（真实执行 + 事务回滚 / 真实写文件后删 / 真实发 Kafka 不消费 / 完整全链路副作用沙箱）；
- "实跑后回滚"作为 dry-run 替代（业务表 + 外部 IO 已改，回滚不可靠）；
- mixed mode（同一 instance 内一部分 step dry 一部分 real）；
- 复用 bypass-mode 开关（语义不同：bypass 是放行不安全，dry-run 是安全但不副作用）；
- v1 不做 dry-run vs real diff 工具（用 metric / audit 各自看）。

**判定边界的提问**：「这个 dry-run 想看什么？」"看配置对不对 / 看会不会跑 / 看 SQL 能不能跑" → 属本 ADR；"跑出来看业务结果对不对" → 不属本 ADR（那是 UAT 环境的活）。

---

### ADR-027 资源亲和 / 地理调度（P2/P3，暂缓）

✅ **当前保留的轻量字段（非本 ADR，是基线）**：
- `worker_group`（已有）；
- `workerType` / `capabilityTags` / `resourceProfile` / `region` / `zone` / `dataAffinityKey` 字段预留即可；
- 简单策略：`workerType` 必匹配 → `capabilityTags` 必满足 → `region/zone` 优先匹配 → 不做复杂打分。

✅ **触发条件到了再做完整版**：
1. worker 部署 ≥ 2 个机房，且业务有数据本地性 / 容灾倾斜；
2. 异构硬件（GPU / FPGA / 高内存 / 低内存）；
3. 合规要求"风控 worker 物理隔离普通 ETL"；
4. license 绑机（Oracle 客户端 / Db2 client）；
5. `worker_group` 数 ≥ 8 且仍增长。

❌ **绝不做（系统定位红线）**：
- 自研 Kubernetes Scheduler / 多集群调度平台 / 容器资源编排器；
- PodAffinity / TopologySpreadConstraints / 节点拓扑调度；
- 底层机器资源编排 / 节点生命周期管理；
- 复杂 multi-objective optimization 调度算法；
- label 表达式（regex / 计算字段）—— K8s 也只支持 In/NotIn/Exists/DoesNotExist 四种。

**判定边界的提问**：「这个调度策略是在挑 worker 还是在挑机器？」挑 worker → 属本 ADR；挑机器 / 挑容器 / 挑节点 → 不属本 ADR（那是 K8s 的活，本系统不重做）。

## 4. 实施触发信号矩阵

按 production 信号触发优先级抬升 / 启动暂缓项：

| 信号 | 行动 |
|---|---|
| 凌晨配置事故"workflow 跑挂才发现配错"复盘 ≥ 1 次 | **启动 ADR-025 静态校验** |
| 一次 DB 闪断把所有 instance 标 FAILED 半夜起来 RERUN | **启动 ADR-012 失败分类** |
| 跨境业务 / 半天 cutoff 诉求 / calendar 数 ≥ 5 | **启动 ADR-023 多日历联动** |
| 出现"我读到了一份 SUCCESS 但是数错的数据"投诉 | **启动 ADR-021 数据对账** |
| 监管来查"3 月 15 日 1:30 时 calendar 是怎样" 找不齐 | **启动 ADR-022 Forensic 取证** |
| 大版本升级 / 新 job 上线 / 灾备演练 诉求出现 | **启动 ADR-026 dry-run 轻量版** |
| `archive.job_instance_archive` 行数 ≥ 5 亿 | **启动 ADR-024 冷热分层** |
| 多机房部署 / 异构硬件 / 合规隔离 诉求出现 | **启动 ADR-027 资源亲和（仅完整版）** |

## 5. 三大越界风险一句话

| ADR | 越界形态 | 防止动作 |
|---|---|---|
| ADR-021 | 扩张为"数据治理 / 财务核算 / 主数据 / 血缘"平台 | "修业务数据 vs 裁定业务对错"二分判定，**只做前者** |
| ADR-026 | 扩张为"完整副作用沙箱 + 事务回滚 + Kafka 不消费"模拟 | 三层粒度 L1/L2/L3 写死，**FULL_SIMULATION 不做** |
| ADR-027 | 扩张为"自研 K8s scheduler + 跨集群编排 + 节点拓扑" | "挑 worker vs 挑机器"二分判定，**挑机器是 K8s 的活** |

---

## 6. 复审节奏

- **每季度复审**：对比触发信号 + production 痛点决定是否启动暂缓项；
- **触发立即复审**：任何一条 §4 信号出现即排议程；
- **死线**：若任何 ADR 标 Accepted 但 12 个月仍未实施，重新评估 boundary（往往是边界写错了 / 触发条件写松了）。
