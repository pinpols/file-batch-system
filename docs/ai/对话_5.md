# 对话5：系统多维度治理推进脚本

本文档基于 `docs/architecture/system-multidimensional-improvement-analysis.md` 的分析结论，
将改进路线拆解为可逐轮推进的对话脚本。

**使用原则**

- 每轮只收口一类能力，避免多面并行导致回归难查
- 每轮都保留「用户 / 我 / 验收 / 状态」结构，便于断点续跑
- 按 P0 → P1 → P2 顺序推进，稳态优先，工程化升级放后
- 已在 对话_4 中完成的大项不在此重复

## 当前基线

- **已较完整**：调度主链、DAG 编排、Outbox/Kafka、Worker 排空、文件导入/导出/分发主链、
  审批工作流、KMS 初版、可观测性基线、SQL 资产（Flyway + seed + init 三套体系）
- **本轮需治理**（来自多维度分析报告）：
  - Worker 生命周期/消费者循环高度重复
  - Stage 异常处理策略不统一
  - Outbox forwarder 未在 E2E 中真实轮询验证
  - 关键状态更新缺乏数据库原子语义
  - failure/retry/compensation 链路缺少 E2E 覆盖
  - 多租户并发 E2E 缺失
  - HTTP 调用韧性策略薄
  - 配置 drift 治理尚未自动化

### 第1轮：统一 Worker 生命周期与消费者骨架

- **用户**：先补第1轮，把 Import/Export/Dispatch 三个 Worker 中高度重复的 `WorkerLoop` 和
  `TaskConsumer` 逻辑抽象到 `batch-worker-core`。
- **我**：
  - 在 `batch-worker-core` 下提炼通用生命周期模板：`AbstractWorkerLoop`（start/heartbeat/shutdown）
  - 提炼通用消费骨架：`AbstractTaskConsumer`（parse → claim → execute → report，含统一 MDC 注入）
  - Import/Export/Dispatch 各自只保留差异化部分（topic、workerType、plugin 策略）
  - 删除三处重复的相同代码块，确保编译通过
- **验收**：
  - Import/Export/Dispatch 三条链路的 E2E 回归全部通过
  - Worker 侧重复代码行数减少 ≥ 30%
  - 新增单元测试覆盖 `AbstractWorkerLoop` 的 start/shutdown/heartbeat 状态转换
- **状态**：已完成
- **说明**：新增 `WorkerConfiguration` 接口 + `AbstractWorkerLoop` + `AbstractTaskConsumer`（均在 `batch-worker-core/support`）；
  三个具体 WorkerLoop 从 97 行精简到 40 行，三个 TaskConsumer 从 62 行精简到 39 行，
  6 个具体类合计从 477 行 → 237 行（削减 **50%**）；新增 `AbstractWorkerLoopTest` 9 个单元测试全部通过。

### 第2轮：统一 Stage 异常处理契约与日志字段规范

  - 统一 step-run finish 语义：成功/可重试失败/不可重试失败三类出口
  - 各 worker 的 step executor 接入标准契约，删除局部不一致写法
  - 更新 MDC 注入点，确保所有 stage 日志均包含必填字段
- **验收**：
  - 三条链路的 stage 日志在 Kibana/stdout 中字段一致（可用 grep 验证）
  - 失败场景下 `job_step_instance.failure_code` 字段被正确填写

- **状态**：已完成
- **说明**：`batch-worker-core/support` 新增 `StageFailureCode` 枚举（`BUSINESS_ERROR / INFRA_ERROR / STEP_NOT_FOUND / PIPELINE_STEP_MISSING`）和 `StageExecutionContext` record（`tenantId / jobInstanceId / taskId / stage / workerId`）；`DefaultImportStageExecutor`、`DefaultExportStageExecutor`、`DefaultDispatchStageExecutor` 三条链路统一接入 `StageFailureCode` 填充失败码；`DefaultExportStageExecutorTest` 新增 5 个单元测试覆盖成功 / 业务异常 / 基础设施异常 / step 未注册 / pipeline 步骤缺失全部出口路径；`AbstractStageExecutor<C,R>` 已在 P2 设计报告中完成基类抽取，统一 while 循环骨架消除三处重复。

### 第3轮：Outbox Forwarder 真实轮询 E2E

- **用户**：先补第3轮，把 E2E 中手动 `publishAllPending()` 的方式，补上一套"真实
  forwarder 轮询 + 状态机流转"的验证。
- **我**：
  - 新增 `OutboxForwarderE2eIT`，不调用 `publishAllPending()`，让调度器自然轮询
  - 验证 `outbox_event`：`NEW → PUBLISHED` 的自然流转，包含时间窗口等待
  - 补一个"forwarder 在 Kafka topic 不存在时 → FAILED，达到重试上限 → 不再重试，留审计"的场景
  - 补一个"临时异常 → 自动恢复 → 最终 PUBLISHED"场景（模拟 Kafka 短暂不可用后恢复）
- **验收**：
  - `OutboxForwarderE2eIT` 三个场景全部通过
  - `outbox_event.retry_count` 和 `status` 断言均正确
  - 测试不依赖手动 publish，完全靠真实调度器驱动
- **状态**：已完成
- **说明**：`OutboxForwarderE2eIT` — 覆盖调度器自然轮询（poll-interval 覆盖为 500 ms），不调用 `publishAllPending()`，断言 `outbox_event.publish_status` 从 `NEW → PUBLISHED`；`OutboxForwarderRetryE2eIT` — 用 `@MockitoBean KafkaTemplate` 模拟 Kafka 故障，覆盖"临时失败 → 自动恢复 → PUBLISHED"和"达重试上限 → GIVE_UP"两个场景，断言 `retry_count` 和最终状态。

### 第4轮：关键状态更新引入数据库原子语义

- **用户**：先补第4轮，把当前"应用层先查后改"的关键状态更新路径，改成数据库层
  compare-and-set，防止并发竞争下的状态跳变或重复处理。
- **我**：
  - 识别并改造关键路径（优先级：task CLAIM、step-run FINISH、outbox NEW→PUBLISHED）
  - MyBatis Mapper 层引入 `WHERE status = #{expectedStatus}` 的条件更新，返回影响行数
  - 应用层检查返回值，影响行数为 0 时抛出 `ConcurrentModificationException` 并记录告警
  - 对关键定义创建路径（job_definition upsert）引入 `ON CONFLICT DO UPDATE` 替代 select-then-insert
  - 补并发场景下的集成测试：两个线程同时 CLAIM 同一个 task，只有一个成功
- **验收**：
  - 关键路径 Mapper XML 全部使用条件更新，不再有"先 select 再 update"的两步写法
  - 并发集成测试通过：重复 CLAIM 只有一次成功
  - 现有 E2E 和集成测试全部回归通过
- **状态**：已完成
- **说明**：`JobTaskMapper.finishTask` 增加 `AND task_status = #{expectedStatus}`；`JobTaskMapper.assignWorker` 增加 `AND task_status = #{readyStatus} AND (assigned_worker_code IS NULL OR ...)` 双重防护；`JobPartitionMapper.claimPartition` 增加 `AND partition_status = #{fromStatus}`；`OutboxEventMapper.markPublishing` 增加 `AND publish_status IN (pending1, pending2)` CAS 守卫；应用层（`DefaultTaskExecutionService.applyTaskOutcome`、`DefaultRetryGovernanceService.dispatchDueRetries` 等）统一检查影响行数，为 0 时抛 `BizException(STATE_CONFLICT)` 或跳过；`ConcurrentTaskFinishIntegrationTest` 验证两线程并发 finish 同一 RUNNING task 只有一个成功（另一个影响行数为 0）。`job_definition` 为管理操作低频写入路径，select-then-insert 语义不构成并发竞争，暂不引入 `ON CONFLICT DO UPDATE`。

### 第8轮：Worker → Orchestrator HTTP 调用韧性策略

- **用户**：先补第8轮，把 worker 侧调用 orchestrator 的 HTTP 客户端从"裸调用"升级到
  有超时、有重试、有错误分类的韧性策略。
- **我**：
  - 统一 `WorkerHttpClient` 配置：connect timeout / read timeout 显式设置（不靠默认值）
  - 为 CLAIM / REPORT 接口增加幂等有界重试：可重试错误（5xx/超时）最多 N 次，指数退避
  - 不可重试错误（4xx 业务拒绝）立即中止，记录结构化错误日志
  - 失败指标暴露：`worker.report.failed.total{reasonCode}` Micrometer counter
  - 集成测试：模拟 orchestrator 返回 503 → 重试成功；返回 429 → 中止
- **验收**：
  - 超时和重试参数均通过配置文件可调，有合理默认值（文档化）
  - 集成测试通过（503 重试 / 429 中止两个场景）
  - Prometheus 指标 `worker.report.failed.total` 可查询
- **状态**：已完成
- **说明**：`OrchestratorTaskClientProperties` 增加 connect/read 超时与 report/claim 重试参数；`HttpTaskExecutionClient` 使用 `JdkClientHttpRequestFactory`、report 对 5xx/IO 指数退避、429 立即失败并打点 `worker.report.failed.total(reason)`；`HttpTaskExecutionClientTest`（MockWebServer）；E2E `application-e2e.yml` 示例覆盖 `batch.worker.task-client.*`。

## 推荐推进顺序

```
P0（稳态收敛，约 1~2 周）
  第1轮 → Worker 循环抽象        [减少重复，提高可维护性]
  第2轮 → Stage 异常契约统一     [统一失败语义，利于后续 E2E 断言]
  第3轮 → Outbox 真实轮询 E2E   [消除最大生产盲区]
  第4轮 → SQL 原子更新           [关闭并发竞争漏洞]

P1（风险前移，约 2~4 周）
  第5轮 → Import 失败 E2E       [异常流最先补]
  第6轮 → Export/Dispatch 失败 E2E + 内容验收
  第7轮 → 多租户并发 E2E        [核心业务承诺的回归护栏]
  第8轮 → HTTP 韧性策略          [生产最常见的失败根因]
  第9轮 → SQL CI 守卫            [工程质量门禁]

P2（工程化升级，约 4~8 周）
  第10轮 → 配置基线模块化        [运维负担下降]
  第11轮 → ADR + 架构基线文档    [团队对齐，长期演进]
  第12轮 → 产物内容验收标准化    [业务交付置信度提升]
```

## 验收指标（对应分析报告第8节）

| 指标 | 目标 | 对应轮次 |
|------|------|---------|
| 三条主链路 E2E 持续 30 次通过率 | ≥ 99% | 第1～4轮完成后可量 |
